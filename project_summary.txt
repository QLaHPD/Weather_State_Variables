Weather_State_Variables Project Summary
Generated: 2026-04-19

This file summarizes the main engineering work completed so far in this repo.
It is meant to be a practical snapshot of the current state of the project, not
an exhaustive changelog.

1. Data pipeline and loading

- Reworked the dataloader/prefetch path so sequential training can keep data in
  RAM more effectively and reduce NVMe stalls.
- Added ring-buffer style dynamic caching logic for time windows so batches can
  be overwritten in RAM as they are consumed.
- Preserved sequential locality for weather time series instead of breaking
  overlap reuse across many workers.
- Current goal of this pipeline is to keep sample loading smoother and reduce
  the bursty storage behavior that was throttling throughput after a few
  hundred batches.

2. Data normalization

- Added normalization support for the dynamic weather variables.
- Added denormalization helpers so validation and plotting can report metrics in
  physical units instead of only normalized space.
- Validation now reports overall loss plus per-variable losses and denormalized
  MAE statistics.
- The dataset can ensure normalization statistics exist before evaluation.

3. Main model training improvements

- Added checkpoint saving every N train batches and every N optimizer steps.
- Added resume-training support from checkpoints.
- Resume logic restores model, optimizer, scaler, history, batch counters,
  optimizer-step counters, and epoch state.
- Resume compatibility was relaxed so learning rate, weight decay, and gradient
  accumulation overrides can be changed when resuming.
- Batch counters now persist globally instead of being treated as epoch-only
  information.

4. Main model architecture

- Confirmed and then enforced a true encoder/decoder bottleneck split in the
  main forecast model.
- The decoder was changed so it depends on the bottleneck representation rather
  than relying on extra skip-style shortcuts across the split.
- The temporal embedding (temb) remains algorithm-generated metadata, not an ERA
  source variable.

5. Main model validation and rollout tooling

- Added standalone main-model validation script:
  scripts/validate_main.py
- Added rollout plotting and image export support.
- Validation can save rollout plots with real input, real target, prediction,
  and denormalized error views.
- Added standalone autoregressive rollout script:
  scripts/rollout_main.py
- Rollout exports each channel to its own folder with filenames like:
  t3h.png, t6h.png, t9h.png, ...
- Rollout supports iterative one-step autoregressive prediction.

6. Main model rollout with intrinsic chaining

- Added optional intrinsic-model chaining into rollout.
- If an intrinsic checkpoint is supplied, rollout can run:
  main encoder -> intrinsic model -> main decoder
- Added intrinsic application frequency control:
  --intrinsic-frequency N
- Example meaning:
  frequency 4 => use intrinsic on rollout steps 4, 8, 12, ...

7. Intrinsic model evolution

The intrinsic model went through several design iterations:

- Initial transformer-based variants were discussed and partially implemented.
- The current intrinsic model is a convolutional autoencoder rather than a
  transformer.
- It reconstructs the main model encoder patch-grid representation.
- Large flatten-then-linear bottlenecks were replaced by convolutional
  bottleneck projection.
- The current architecture uses:
  input projection if needed
  residual conv block
  multiple downsampling stages
  convolution to a 1x1 intrinsic code
  inverse convolution back from the code
  mirrored decoder stages
  output projection

Current intrinsic-model config controls include:

- feature_channels
- resblocks_per_stage
- d_intrinsic
- num_groups
- apply_tanh

Important note:
- The intrinsic model is currently conv-based, not Swin-based and not using
  standard transformer encoder blocks anymore.
- It does not use positional embeddings.

8. Intrinsic training and validation

- Added intrinsic training path using frozen main encoder features as the target
  reconstruction space.
- The intrinsic objective is standard MSE reconstruction of the encoder output.
- The main encoder is frozen, but gradients are allowed to flow through the
  latent path as needed for reconstruction loss computation.
- Added standalone intrinsic validation script:
  scripts/validate_intrinsic.py
- Validation supports a "chain" mode:
  main encoder -> intrinsic -> main decoder
- This chain mode can produce the same kind of forecast-style plots for
  qualitative inspection.

9. Forecast/data semantics clarified

Clarified config behavior for:

- input_time_offsets_hours
- lead_time_hours
- forecast_steps
- sample_stride_hours

Example:

input_time_offsets_hours: [-3, 0]
lead_time_hours: 3
forecast_steps: 2
sample_stride_hours: 1

This means:

- input frames are at -3h and 0h
- targets are +3h and +6h
- anchors advance every 1 hour through the dataset

10. Scaling-law and model size support

- Added utilities to instantiate the model on CPU and inspect parameter count.
- Used this to reason about model scale versus number of training tokens/patches.
- This was intended for scaling-law sanity checks without occupying GPU memory.

11. Intrinsic-dimension estimation

This area saw the most recent development.

11.1 Initial estimator

- Implemented Levina-Bickel intrinsic-dimension estimation for the main model
  latent.
- The main latent source defaults to the output of the second transformer block.
- Latents are flattened to vectors before estimation.

11.2 Two-NN estimator

- Added Two-NN as an alternative and later made it the default method.
- Implemented the Two-NN fit using the r2/r1 ratio and origin-constrained line
  fit from the paper.
- Added configurable tail trimming:
  --two-nn-discard-fraction

11.3 Effective-ID plateau search

- Added automated plateau search to estimate a relevant/effective intrinsic
  dimension for learning rather than only a single full-pool estimate.
- Plateau search uses one cached latent pool and does not regenerate samples for
  each subset size.
- If max_samples is 1024, the code generates those 1024 samples once and then
  uses random subsets from that same pool for sizes like 128, 256, 512, 1024.
- Added automatic doubling schedule and optional explicit subset-size override.
- Plateau output includes a recommended_effective_dimension.

11.4 Multi-GPU intrinsic-dimension estimation

- Added support for distributed latent generation with torchrun.
- batch_size is interpreted as per-GPU batch size.
- effective batch size is now:
  batch_size * world_size
- A fixed shard sampler avoids padding and duplicate sample generation across
  ranks.
- Rank 0 gathers latent shards and performs the final intrinsic-dimension
  calculation.

11.5 Progress reporting

- Added a progress bar for latent-sample generation.
- In multi-GPU mode the progress reflects the total global sample count.
- JSON mode suppresses the progress bar automatically.
- Manual disable flag:
  --no-progress

12. Current behavior of intrinsic-dimension estimation

Current execution stages:

1. GPU phase:
   generate latent samples using the main encoder
2. CPU phase:
   gather latent pool on rank 0
   compute full pairwise distance matrix
   run Two-NN or Levina-Bickel
   run plateau search if requested

Important caveat:

- The actual distance-matrix calculation for Two-NN / Levina-Bickel is still
  CPU-side on rank 0.
- For large max_samples (for example 4096) this can take a long time after the
  progress bar reaches 100%.
- This is not a deadlock by default; it is usually the expensive pairwise
  distance step.

13. Distributed runtime fixes

- Fixed NCCL device binding warnings by explicitly binding init_process_group to
  the current CUDA device when supported.
- Added a helper so distributed barriers use the correct CUDA device id.
- This makes the multi-GPU intrinsic-dimension path safer and less likely to
  hang because of ambiguous device mapping.

16. Current open caveats / next useful improvements

- The intrinsic-dimension distance computation is still CPU-bound and may need
  chunking or a GPU-backed implementation for very large sample counts.
- A second progress indicator for the pairwise-distance / plateau stage would
  make long runs easier to interpret.
- Saving a plateau plot (dimension estimate vs sample size) would make the
  effective-ID selection easier to inspect visually.

17. Practical interpretation of recent intrinsic-dimension results

- The single-shot full-pool Two-NN estimate can be notably higher than the
  plateau-based effective dimension.
- For learning, the plateau-based recommended_effective_dimension is the more
  relevant quantity.
- If the full-pool estimate is much larger than the plateau value, that usually
  suggests finer-scale or noisier directions become visible at larger sample
  counts.

End of summary.
